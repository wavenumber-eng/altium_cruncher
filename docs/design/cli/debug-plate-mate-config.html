debug-plate portable mate config design

Status

This document is a design draft for the next config shape. The implemented command contract remains debug-plate.html, and the generated Altium MCO operation contract remains mco.html. Current draft implementation covers mate config seeding, selector resolution, mate part roles, PCB labels, and source-side validation; remaining projection actions are still draft.

Goal

The current command name is debug-plate, but the domain model is broader: create an output board that mates to an input board. The first feature-complete use case is cricket-node mated to node-test-array derived fixture parts. The config model should remain portable enough to map later to KiCad and a generic pcb_cruncher workflow, while the current MCO backend remains Altium-specific.

Vocabulary

Term Meaning
source, input, DUT The existing board being analyzed. For the first workflow this is cricket-node.
destination, output, mate The generated board that physically mates to the source board.
human config The human-authored or GUI-authored source of truth. It describes intent and should avoid derived facts.
resolved plan The planner's internal view after reading source board facts: coordinates, nets, sides, source geometry, selected objects, and projection actions.
MCO The derived, inspectable, Altium-specific operation script compiled from the human config and resolved plan.

Config Principles

  • The config is source code; MCO is compiled output.
  • The config should name what to select and what to project, not manually repeat coordinates, net names, or source side in normal use.
  • The planner resolves physical and electrical source facts from the source project.
  • The config should support both hand editing and future GUI authoring.
  • Portable concepts belong in config; backend-specific primitive details belong in generated MCO or backend settings.
  • The first implementation may keep an Altium command and Altium cache, but the conceptual config should not depend on Altium-only object names where a portable term is available.

Theory Of Operations

  1. inspect loads the source project and classifies source objects that are likely useful for mating.
  2. seed writes a reviewable config that uses selectors and projection rules rather than fully resolved object records.
  3. The user or future GUI edits the config to choose source objects, mate parts, graphics, labels, output paths, and artifacts.
  4. plan resolves selectors against the source project, validates source-side and geometry rules, and compiles MCO.
  5. run executes the generated MCO to create the output board and supporting artifacts.
  6. The user opens the output project in Altium for visual inspection, then iterates on the config.

Source Object Classes

The cricket-node use case needs these source object classes to be explicit in the planner, even when the first config uses presets or designator patterns.

Class Resolved Facts First Use
component Designator, footprint, schematic linkage when available, side, center, pads, net names, pin/pad mapping. TP-style test pads, M-style mounts, future J/connector/header mating.
free pad or drill Location, size, hole size, plated state, layer/span, net name when resolvable. Cricket-node 2 mm NPTH alignment pins and other free pads.
board outline Closed outline geometry, origin relationship, arcs/lines when available. Mechanical reference graphics or output outline setup.
internal cutout Regions inside the outline, shape geometry, layer/object metadata. Mechanical reference graphics and optional actual output cutouts.

Projection Actions

Action Meaning
mate_component Place a different symbol/footprint that mates to the selected source object, such as a pogo pin for a source test pad.
copy_source_footprint Project the source footprint 1:1 into the output. This supports .100 inch headers, shield-style boards, and early workflows without a curated mate library.
same_pad Project equivalent pads or drills without a full component abstraction.
reference_graphics Add mechanical or silkscreen reference geometry, such as circular rings around source pads, square outlines, bounding boxes, or future pin-1 markers. The current implemented subset supports circular source_pad_outline rings from inspected source pad geometry.
label Add text labels near mate features or in arranged groups. Labels may be tied to source nets, source designators, or custom text.
breakout Add additional connectors wired to selected source nets, such as future Saleae-style 2x4 headers with generated net labels and text labels.
outline_or_cutout Project source board outline or internal cutouts as graphics, actual output cutouts, or both.
alignment_artifact Produce generated helper outputs such as a bottom-layer STEP artifact from pcb-layer-step.

Side And Origin Rules

  • The planner should infer selected source object side from the source board.
  • The user should not normally specify source_mount_side; it is a resolved fact or optional validation expectation.
  • Selections that mix top-side and bottom-side SMT-like objects should fail with a clear diagnostic.
  • Mechanical mount features may be listed in validation.side_agnostic_kinds when they should not participate in signal-side inference.
  • Through-hole or 1:1 copied connector workflows may be explicitly side-agnostic.
  • For cricket-node bottom-mount style workflows, source and output board origins should remain aligned so STEP artifacts and projected geometry line up.
  • When output.origin is preserve_source, the planner resolves source outline bounds and passes them to the generated output project skeleton rather than requiring the config to repeat derived coordinates.

Cricket-Node Feature-Complete Target

The first complete workflow should support cricket-node without requiring users to type resolved coordinates or net names. The config should select source objects, and the planner should resolve the facts needed to compile MCO.

  • Select cricket-node test points by explicit designator, range, or pattern such as TP*.
  • Select mount features and allow node-test-array M5-M8 normalization to output M1-M4 where needed.
  • Select free NPTH alignment holes as source free-pad/drill objects.
  • Project mate components from the current node-test-array cache.
  • Add configurable right or left side PCB net labels for source nets.
  • Add optional circular reference graphics on a configurable mechanical layer for source pad alignment.
  • Project the source board outline to a configurable graphics layer.
  • Detect and project internal board cutouts as graphics and later actual cutouts.
  • Generate a bottom-layer pcb-layer-step artifact for visual/mechanical alignment.
  • Group generated PCB-side features into DEBUG_PLATE_FEATURES.

Seed And GUI Behavior

Before a GUI exists, seed should produce a commented config with useful selector presets and disabled optional sections. Later, a GUI can render the source board SVG, show grouped source objects such as test points, debug pads, connectors, and mounts, and write the same config format after user selection.

Source selectors are local to each projection group. Component designators may be a wildcard, a compact range such as TP1-5, a comma-separated expression such as TP1-5, U12, or an explicit JSON array for machine-generated configs.

Projection actions own their output options. For example, two projection groups may both use label actions but choose different placement side, offsets, box sizes, and text style. Mate configs therefore do not use the legacy global pcb_labels block as the primary label contract.

Artifact highlights reference projection ids instead of duplicating source selectors. pcb-layer-step currently owns layer, thickness, and base color options; the mate config adds projection-level highlight intent that can compile into that artifact workflow later.

Deferred Topics

  • Full Saleae-style breakout connector automation.
  • Multi-pin connector mate pin mapping and mirrored pin-order policies.
  • Generic update-in-place behavior for existing mate boards.
  • Final naming if the command eventually becomes mate or moves under a generic PCB package.
