CLI command design

mco

Executes Monkey Change Order JSONC operation files for generated CAD automation.

Purpose

mco is the execution substrate for generated workflows such as debug-plate. It keeps automation intent in JSONC while the executor owns CAD document lifecycle: open on first use, keep documents live during the run, and flush dirty documents on exit.

Usage

altium-cruncher mco init debug-plate.mco.jsonc --force
altium-cruncher mco list-ops
altium-cruncher mco run debug-plate.mco.jsonc --dry-run
altium-cruncher mco run debug-plate.mco.jsonc

Arguments

AreaContract
schemaMCO files use wn.altium_cruncher.mco.v1. JSONC comments, trailing commas, and UTF-8 BOM are accepted.
operations[]Ordered operation objects with required id and op, optional args, optional result message, and optional on_fail branch target.
Core opsmco.message, mco.fail, and file.copy.
Project opsproject.create-skeleton creates a minimal Altium PCB project, schematic, and board.
Schematic opsschdoc.add-wire, schdoc.add-net-label, and schdoc.add-component.
PCB opspcbdoc.add-text, pcbdoc.add-component, geometry insertion operations, user-union creation, and pcbdoc.export-layer-step.

Output

run prints a JSON result containing overall success, dry-run state, and one result object per operation. Mutation operations return output paths and operation-specific fields. On a normal run, dirty CAD documents are saved after operation execution completes.

{
  "schema": "wn.altium_cruncher.mco.result.v1",
  "ok": true,
  "dry_run": false,
  "results": []
}

Tests

Coverage lives in tests/test_mco.py. It verifies JSONC parsing, control flow, dry-run handling, CLI init/list/run behavior, document session flushing, schematic and PCB insertion operations, user unions, text style options, library placement, and STEP export operation plumbing.
